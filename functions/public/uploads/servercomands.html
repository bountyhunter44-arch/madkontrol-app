10. Nginx config

nano /etc/nginx/sites-available/madkontrollen-agent
server {
        listen 80;
        listen [::]:80;
        server_name agent.sangmaskine.dk;

        client_max_body_size 25m;

        location / {
                proxy_pass http://127.0.0.1:8000;
                proxy_http_version 1.1;

                proxy_set_header Host $host;
                proxy_set_header X-Real-IP $remote_addr;
                proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
                proxy_set_header X-Forwarded-Proto $scheme;

                proxy_read_timeout 300;
                proxy_connect_timeout 60;
                proxy_send_timeout 300;
        }
}

🌐 Wildcard Subdomæner

*.madkontrollen.dk Setup

Denne sektion viser hvordan du opsætter wildcard subdomæner (fx pizzeria-roma.madkontrollen.dk) der automatisk proxyer til din Firebase Cloud Function seoSiteRenderer.

Wildcard Domæne
*.madkontrollen.dk
Cloud Function
seoSiteRenderer
Firebase Region
us-central1
Function URL
https://us-central1-madkontrollen.cloudfunctions.net/seoSiteRenderer

17. Hetzner DNS Wildcard A-record

✅ Dette har du allerede gjort! Du har oprettet wildcard A-record i Hetzner DNS.

Host: *
Type: A
Value: 46.62.246.141
TTL: 3600

Dette betyder at ALLE subdomæner (test.madkontrollen.dk, pizzeria-roma.madkontrollen.dk osv.) peger på din VPS.

18. Hetzner API Token (til DNS-validering)

For at få wildcard SSL-certifikat skal Certbot kunne validere via Hetzner DNS API.

1

Opret API Token i Hetzner

Gå til: https://dns.hetzner.com/settings/api-token

Klik "Create access token" og gem token'et sikkert.

2

Opret credentials-fil på serveren

mkdir -p /root/.secrets/certbot
nano /root/.secrets/certbot/hetzner.ini

Indsæt følgende (erstat DIT_HETZNER_API_TOKEN):

dns_hetzner_api_token = DIT_HETZNER_API_TOKEN

Sikr filen:

chmod 600 /root/.secrets/certbot/hetzner.ini

19. Installér Certbot DNS Plugin

apt install -y certbot python3-certbot-dns-hetzner

Hvis pakken ikke findes, installér via pip:

apt install -y python3-pip
pip3 install certbot-dns-hetzner

20. Få Wildcard SSL Certifikat

certbot certonly \
  --dns-hetzner \
  --dns-hetzner-credentials /root/.secrets/certbot/hetzner.ini \
  -d madkontrollen.dk \
  -d *.madkontrollen.dk \
  --agree-tos \
  --non-interactive \
  --email din@email.dk

Certifikatet gemmes i: /etc/letsencrypt/live/madkontrollen.dk/

Husk at erstatte din@email.dk med din rigtige email!

21. Nginx Wildcard Config

Opret en ny Nginx config der håndterer ALLE subdomæner og proxyer til Firebase Cloud Function.

nano /etc/nginx/sites-available/madkontrollen-wildcard
# HTTP → HTTPS redirect
server {
    listen 80;
    listen [::]:80;
    server_name *.madkontrollen.dk madkontrollen.dk;

    return 301 https://$host$request_uri;
}

# HTTPS Wildcard Proxy
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name *.madkontrollen.dk madkontrollen.dk;

    # SSL Certifikat
    ssl_certificate /etc/letsencrypt/live/madkontrollen.dk/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/madkontrollen.dk/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    # Proxy til Firebase Cloud Function
    location / {
        proxy_pass https://us-central1-madkontrollen.cloudfunctions.net/seoSiteRenderer;
        proxy_ssl_server_name on;
        proxy_http_version 1.1;

        # Videresend Host header (vigtigt!)
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts
        proxy_read_timeout 60;
        proxy_connect_timeout 60;
        proxy_send_timeout 60;
    }
}

22. Aktivér Nginx Wildcard Config

26. Sådan virker flowet

1. Kunde besøger: pizzeria-roma.madkontrollen.dk
   ↓
2. DNS wildcard → 46.62.246.141 (din VPS)
   ↓
3. Nginx modtager request med Host: pizzeria-roma.madkontrollen.dk
   ↓
4. Nginx proxy til Firebase Cloud Function (seoSiteRenderer)
   ↓
5. Cloud Function læser Host header
   ↓
6. Cloud Function slår op i Firestore (fx companies collection)
   ↓
7. Cloud Function returnerer SEO-optimeret HTML
   ↓
8. Nginx sender HTML tilbage til kunde
Din seoSiteRenderer skal læse Host header for at vide hvilket subdomæne der blev kaldt!

💳 Stripe

Stripe Produktion, Test og Webhook

Denne sektion samler den praktiske opsætning af Stripe i Madkontrollen Pro med Firebase Functions, secrets, webhook og anbefalet test/live-struktur.

Stripe Function URL
https://us-central1-madkontrollen.cloudfunctions.net/stripeWebhook
Backend runtime
Firebase Functions v2, Node.js 22
Secrets
STRIPE_SECRET_KEY og STRIPE_WEBHOOK_SECRET
Vigtigt
Secret keys må aldrig ligge i frontend
Både sk_live_... og whsec_... er backend-hemmeligheder. De må aldrig ind i HTML, browser-JS eller offentlige konfigurationsfiler.

29. Stripe nøgler — hvad er hvad?

Publishable key
pk_live_... / pk_test_...
Må bruges i frontend.
Secret key
sk_live_... / sk_test_...
Kun backend. Bruges til Stripe API-kald.
Webhook secret
whsec_...
Kun backend. Bruges til at verificere Stripe webhook events.
Vigtig regel
Bland aldrig test og live
Test keys virker kun med testdata. Live keys virker kun med rigtige betalinger.

30. Korrekt webhook endpoint

Stripe webhook skal pege på din Firebase Cloud Function URL — ikke på .web.app.

https://us-central1-madkontrollen.cloudfunctions.net/stripeWebhook
Forkert eksempel: https://us-central1-madkontrollen.web.app.cloudfunctions.net/stripeWebhook
Rigtigt eksempel: https://us-central1-madkontrollen.cloudfunctions.net/stripeWebhook

31. Events der skal vælges i Stripe Webhook

Vælg kun de nødvendige events:

checkout.session.completed
customer.subscription.created
customer.subscription.updated
customer.subscription.deleted
invoice.paid
invoice.payment_failed

32. Sæt Stripe secrets i Firebase

Kør disse kommandoer i projektet. Det er backend-hemmeligheder.

firebase functions:secrets:set STRIPE_SECRET_KEY
firebase functions:secrets:set STRIPE_WEBHOOK_SECRET
Når du bliver spurgt, indsætter du henholdsvis sk_live_... eller sk_test_... og derefter whsec_....

33. Deploy er påkrævet efter secrets

Når secrets er sat eller ændret, skal functions deployes igen.

firebase deploy --only functions

Deploy er påkrævet her, fordi Cloud Functions først får secrets tilgængelige efter deploy.
